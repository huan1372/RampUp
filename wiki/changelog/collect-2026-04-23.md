## Key Findings

### 1. TurboQuant Now Uses FA3/FA4 on Hopper and Blackwell (PR #40092, April 23)

**Impact: High.** TurboQuant's prefill paths previously defaulted to FlashAttention 2 regardless of hardware. This was a silent performance regression on all Hopper (H100/H200/H20, SM90) and Blackwell (SM100) deployments.

PR #40092 adds FA version detection to `flash_attn_varlen_func` calls in TurboQuant's prefill path. On NVIDIA H20 (SM90):

| Workload | Throughput improvement | TTFT reduction |
|----------|------------------------|----------------|
| prefill_heavy | +71–89% | −43–54% |
| long_balanced | +46–58% | −62–63% |

Secondary fix: `_get_sliding_window_configs()` now skips non-FlashAttention layers rather than asserting all layers are `FlashAttentionImpl`. This was causing crashes for models that mix TurboQuant quantized layers with standard FlashAttention layers. `--enforce-eager` flags removed from TurboQuant eval configs as FA3 now works with CUDAGraph.

This fix suggests that all TurboQuant benchmarks published before April 23 on Hopper hardware understate actual throughput — the published numbers were using FA2.

### 2. NVFP4-Quantized Checkpoints Now Run on H100 and AMD (PR #35737, April 22)

**Impact: Medium.** NVFP4-format checkpoints (e.g., Qwen3-30B-A3B-NVFP4) were previously hard-gated to Blackwell hardware. PR #35737 adds a Triton-based software emulation backend that enables these checkpoints to load and run on NVIDIA H100 (SM90) and AMD MI300/MI350.

Emulation accuracy cost is small: ~3.3% word perplexity increase (10.90 → 11.26 on Qwen3-30B-A3B-NVFP4). PIQA accuracy stable at 0.79–0.80.

Practical implication: H100 clusters (dominant cloud hardware in 2026) can now load Blackwell-targeted NVFP4 model releases without requiring rebasing to BF16. Memory footprint benefit (NVFP4 checkpoint vs BF16) is preserved. Compute throughput is lower than Blackwell native since operations are emulated in Triton. For H100 operators, this reduces checkpoint compatibility maintenance overhead.

### 3. Warm Compile Time Reduced 88–96% (PR #40151, April 23)

**Impact: High for production deployments.** vLLM's warm compilation path (reloading a previously-compiled model after restart) previously involved deserializing FX graph representations at runtime. PR #40151 eliminates this by inlining attention submodules as Python code and using that directly as the runtime truth.

| Model | Before (s) | After (s) | Reduction |
|-------|-----------|----------|-----------|
| DeepSeek-V3.2 | 6.05 | 0.27 | −95.5% |
| GLM-4.7-FP8 | 7.07 | 0.46 | −93.5% |
| GPT-OSS-120B | 1.57 | 0.19 | −87.9% |
| Llama-3.3-70B | 3.95 | 0.20 | −94.9% |
| Qwen3.5-35B | 2.91 | 1.36 | −53.3% |

Sub-2-second warm compile times across all major models except Qwen3.5-35B (1.36s). For production auto-scaling workflows that restart workers under load, this reduces the "compilation dead time" from up to 7 seconds to under 0.5 seconds.
