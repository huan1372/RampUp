## Key Findings

### 1. vLLM v0.20.0: Largest Release in the Knowledge Base (April 23, 2026)

**Impact: High.** 546 commits from 257 contributors. This is not a patch release — it is a generational update touching every major subsystem.

**Dependency shift**: PyTorch 2.11 + CUDA 13.0 as defaults is a hard environmental requirement. Operators on CUDA 12.x must plan upgrades. This is the first major CUDA version bump since vLLM adopted the rapid-CUDA cadence.

**Key inference-optimization additions**:

1. **FA4 as default MLA prefill backend (SM90+)**: corrects the regression where TurboQuant/MLA paths defaulted to FA2 on Hopper. Combined with PR #40092 (FA3/FA4 fix for TurboQuant, April 23), all Hopper deployments running MLA models are now on the correct FA version.

2. **TurboQuant 2-bit KV** (4× capacity): extends the quantization spectrum below 3-bit for the first time in a numbered release. No quality benchmarks published yet — caution warranted before production use.

3. **Per-token-head INT8/FP8 KV**: finer quantization granularity. Expected quality improvement over per-tensor FP8 with no memory overhead change.

4. **Online quantization frontend**: eliminates the "pre-quantized checkpoint" requirement. Operators can now load BF16 models and quantize at serving time. Lowers the barrier to quantization adoption.

5. **vLLM IR skeleton**: early-stage but signals the intended path toward Helion-based custom kernel generation as a first-class workflow.

6. **RayExecutorV2 + AOT batch-invariance**: two separate improvements to distributed execution and compilation that reduce steady-state serving overhead.

7. **3FS KVConnector**: first file-system-backed KV connector. Targets Bytedance's internal 3FS storage; demonstrates the KV Connector API is extensible to non-network transports.

8. **CPU draft-model spec decode**: opens speculative decoding to operators with constrained GPU memory. Draft quality and throughput limitations still unknown.

**Breaking changes**: The PyTorch 2.11 + CUDA 13.0 default, metrics API rework (PrefillStats replacing old counters), and pooler config renames are the highest-impact breaks for operators who have monitoring instrumentation built on the old API.

### 2. DeepSeek V4: First Production 1M-Context Open Model (April 24, 2026)

**Impact: Very High — architectural significance.** DeepSeek V4-Pro and V4-Flash achieve 1M-token context at dramatically reduced compute and KV cost through a new hybrid attention paradigm (CSA + HCA + mHC). This is not an incremental update to V3.2 — it introduces fundamentally new attention mechanisms that vLLM had to support from scratch.

**CSA (Compressed Sparse Attention)**: hierarchical access pattern with learned Lightning Indexer retrieval. Analogous to L1/L2/archive memory hierarchy in hardware. The key question is whether the Lightning Indexer quality (does it retrieve the right blocks?) holds across diverse task types.

**HCA (Heavily Compressed Attention)**: 128-token-per-entry fixed compression. Very aggressive — one entry per 128 tokens means the model only "sees" ~7,800 entries for 1M context at this layer. Suitable only for global-context layers.

**mHC**: mathematical constraint (Birkhoff polytope) on the residual connection that prevents activation explosion at extreme depths. Theoretically principled; practical training overhead unknown.

**Efficiency numbers**: V4-Pro at 10% of V3.2's KV cache at 1M context means serving a 1M-context request is comparable in KV footprint to serving a ~100K-context V3.2 request. This is a step-change in long-context serving economics.

**Hardware independence**: Apache 2.0 + Huawei Ascend deployment means zero CUDA dependency. This is the first frontier-scale model confirmed deployable on a non-NVIDIA/AMD stack at full scale.

### 3. Post-v0.20.0 PRs: Sync Elimination and Observability

**PR #40654** (seq_lens_cpu sync removal) is notable: GPU→CPU synchronizations in the speculative decode pipeline are pipeline-bubble sources. Their removal is part of the systematic "zero CPU-GPU sync" program in MRV2. Seeing this addressed post-v0.20.0 confirms the program is ongoing.

**PR #40662** (unified acceptance rate metric) is an observability fix that makes spec decode benchmarking more reliable — important for evaluating P-EAGLE, CSD, and SpecGuard performance claims across engine versions.

**PR #40664** (Qwen3 MoE double-gate) is a correctness+performance fix: double gate computation doubled router FLOPs. For Qwen3 MoE models, this reduces compute overhead with no model output change.
