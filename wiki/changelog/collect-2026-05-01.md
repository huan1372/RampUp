## Key Findings

### vLLM Post-v0.20.0 Kernel Push on Blackwell (PR #41326)

PR #41326 "Faster per-token FP8 group quant packed kernel for Blackwell" continues vLLM's aggressive Blackwell optimization track. The "packed" fusion (FP8 quantization + GEMM in one kernel) eliminates an HBM round-trip between the two ops. This follows PR #38877 (MLA+Group FP8 fusion, April 22), which fused FP8 into MLA attention. Together these suggest a systematic effort to eliminate all unbatched FP8 quantization kernel launches on the Blackwell GEMM path — consistent with the vLLM issue #35792 roadmap.

No benchmark numbers captured. Impact is expected on Blackwell B200/GB200 deployments running FP8 per-token group quantized models (e.g., DeepSeek-class MoE, Qwen3 variants).

### async TP + FP8 Correctness + Performance (PR #39505)

PR #39505 resolves a correctness issue (GitHub #27985) that caused FP8 models on TP≥2 to produce incorrect output or miss async TP overlap when using `torch.compile`. This is both a correctness fix and a performance fix: the async TP overlap was previously broken for FP8, meaning FP8 multi-GPU inference ran slower than FP8 single-GPU for no good reason. The fix enables the overlapped schedule that async TP is designed for.

**Practical impact**: Any production vLLM deployment using FP8 + TP≥2 + `torch.compile` (standard for B200/GB200 multi-node) was likely affected. After PR #39505, FP8 async TP overlap should work correctly.

### HMA KV Offload Series Matures to SWA Support (PR #41228)

PR #41228 (part 12/N) extends the HMA KV offload scheduler to handle sliding window attention groups. The series has been building toward supporting DeepSeek V4's complex multi-group KV layout (CSA+HCA+mHC) with KV offloading. Adding SWA group support is a critical piece: many production architectures mix full attention layers (infinite receptive field) with SWA layers (bounded window), and the offload scheduler must know the window boundary to correctly compute which blocks are offloadable. This is the first time such hybrid architectures can use vLLM's KV offloader correctly.

### CapKV: Principled KV Eviction Replaces Attention-Score Heuristics (arXiv 2604.25975)

The Information Bottleneck formulation in CapKV is significant because it provides a common theoretical frame for comparing all KV eviction methods. The key insight is that attention-score accumulation (H2O, SnapKV) measures *historical query relevance* — a proxy for "was this token attended to in the past?" — not *future information capacity*. These proxies diverge at long contexts (32K+), explaining why heuristic methods degrade under aggressive compression for long sequences.

The leverage-score approximation is O(n) per step, making it practically deployable in a production eviction policy. The main gap before vLLM integration is that vLLM's PagedAttention block manager evicts at the block level (groups of 16–32 tokens), while CapKV's leverage scores are computed per token. Block-level approximation (max or mean leverage per block) would be needed.

**Relationship to existing KB**: CapKV fills a gap — the KB had GRACE (channel-level pruning), TurboQuant/IsoQuant (quantization of retained entries), and TTKV (temporal tiering without eviction), but no principled token-level eviction method. CapKV is the first theoretically grounded one.

### KVShot: Diagnosing the Draft Quality Gap (arXiv 2604.26412)

KVShot is particularly valuable as a diagnostic because it separates two failure modes that the existing KB conflated:

1. **False rejections** (CSD addresses): semantically correct tokens rejected for lexical mismatch
2. **Long-range drift** (KVShot addresses): draft model's context representation decays across speculative steps

These are independent failure modes: a drafter can have high K=1 acceptance but poor K>3 acceptance (drift problem), or high overall acceptance but many semantically-correct rejections (false-rejection problem).

**Connection to PayPal EAGLE3 study**: The PayPal commerce workload showed stable 35.5% acceptance at gamma=3 but diminishing returns at gamma=5. KVShot's finding (drift begins at high k) is consistent: gamma=5 requests draft step indices k=4,5 which are precisely the drift zone. For structured commerce outputs (predictable), even the drifted hidden state may still guide correctly at k=4,5; for reasoning tasks, the drift would be more damaging. This partially explains why gamma=3 is the universal sweet spot across structured and reasoning workloads.
