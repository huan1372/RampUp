## Key Findings

### 1. TurboQuant Long-Context Memory Crisis Resolved (PR #40941)

This is the most significant finding of the run. TurboQuant was practically unusable at long contexts: 57.6 GB of GPU memory consumed by dequantization buffers at 1M tokens (32B, TP=4). This was a **show-stopper for long-context TurboQuant inference**.

PR #40941 moves buffer allocation from per-layer to a global WorkspaceManager pool (one layer's worth reused at a time). The savings scale with context length and layer count — exactly the scenario where TurboQuant's compression is most valuable. As a bonus, fixed-size pre-allocated buffers unlock CUDA Graph capture, which was previously blocked by variable-size per-layer allocations.

Memory savings at 128K context (7.4 GB) are immediately relevant for production MoE serving, where 128K+ prompts are now commonplace for reasoning models like DeepSeek V4. At 1M context, the 57.6 GB savings mean the difference between OOM and a functional deployment.

**This is the completion of the TurboQuant production readiness story**: the sequence is PR #38479 (merge, April 15) → PR #40194 (WHT overhead reduction, April 17-19) → PR #40092 (FA3/FA4 prefill, +71–89% throughput, April 23) → v0.20.0 (numbered release, April 23) → PR #40941 (long-context memory, April 27).

### 2. FP8 Extends to ViT Encoders — Resolution-Dependent Trade-off (PR #38065)

FP8 reaches multimodal models: Qwen3 VL family ViT encoder attention now runs FP8 via FlashInfer cuDNN. The resolution-dependent performance pattern is notable: HD resolution actually regresses (0.87×) while 4K delivers 1.18× on GB200. This makes FP8 ViT attention a conditional optimization requiring resolution-aware configuration.

The core cuDNN kernel speedup of 1.42× on GB300 suggests Blackwell next-gen hardware (GB300, successor to GB200) has better FP8 cuDNN support. The GB200 vs GB300 gap (1.12× vs 1.42×) will matter as GB300 clusters become available.

**Implication for multimodal serving:** Production deployments serving mixed-resolution workloads must either cap FP8 ViT to high-resolution requests or accept the HD regression. vLLM does not yet have per-resolution routing for ViT precision.

### 3. SWA Scheduler Deadlock — Silent Correctness Bug (PR #40946)

The SWA admission bug is a classic "formula drift" correctness issue: two independently maintained code paths implemented the same estimate differently. The startup sizer was correct; the runtime gate was not. On models like Mistral or Gemma with SWA, any deployment that used the minimum KV pool size would deadlock on long prompts.

This bug would be hard to diagnose — the symptoms (requests perpetually refused) look like a resource exhaustion problem, not a formula mismatch. The fix (single source of truth method) is the right engineering solution.

### 4. KV Offload for Remote Decode Completes P/D CPU Transport Path (PR #40346)

PR #40346 is small but strategically important. The vLLM P/D infrastructure now has two transport paths for KV migration:
1. **Direct GPU-GPU** (NVLink, RDMA via Nixl): high-bandwidth, low-latency, B200/NVLink deployments
2. **CPU-backed export** (PR #40346 + OffloadingConnector): lower bandwidth, CPU-DRAM as intermediate staging, useful for heterogeneous clusters or nodes without direct NVLink

With PR #40346, the CPU transport path is complete. The `do_remote_decode=True` flag triggers full-state export; local connectors auto-detect cached blocks and skip re-copy (no penalty on the local path). This is the same pattern HMA multi-group KV offload (PR #39403) requires.

### 5. DeepSeek V4 SiLU Clamp — Accumulating Model Fixes

PR #40950 is the third post-release fix for DeepSeek V4 after DSML streaming (PR #40806, April 26) and HMA multi-group KV offload series (PR #39403, April 25). The SiLU clamp addresses numerical stability in the shared expert — a scenario that may only surface at extreme activation magnitudes (rare in typical inference, but possible with adversarial or long-tail inputs). The kernel-level implementation (clamp baked into `silu_and_mul`) is cleaner than a separate pass.

The accumulation of post-release fixes suggests V4's complexity (CSA + HCA + MLA + MoE + shared expert + streaming tool calls) exceeded what could be fully validated before day-0 launch.
