## Key Findings

### 1. EPLB Has Had a Silent Correctness Bug Since Introduction

PR #40810 reveals that vLLM's EPLB router had a deterministic load-collapse bug: when `top_k` (active experts per token) is a multiple of the replica count — a common configuration (e.g., `top_k=2` with 2 replicas, `top_k=8` with 4 replicas) — the hash function mapped all tokens for those experts to the same replica. This caused >90% load imbalance in adversarial cases, effectively making EPLB non-functional for balanced routing.

**Significance:** EPLB was introduced in v0.19.0 and has been accumulating production deployments. Any deployment using standard MoE top_k values with EP ≥ 2 may have been running with near-degenerate load distribution. The Knuth multiplicative hash fix brings the max/mean ratio to 1.07 — approximately 1.6× better than before on tested configurations. This should be a priority update for production EPLB deployments.

### 2. Humming Kernel: Sub-INT8 Inference Now Native in vLLM

PR #34556 adds the Humming JIT quantization library, which is the first vLLM-native path for W1–W5 (sub-8-bit integer) weight quantization with CUDA graph support. Previous backends (Marlin: W4/W8; CUTLASS: FP4/FP8) had coverage gaps below W4. Humming's JIT model means new quantization formats can be added without writing new CUDA kernels — the library generates kernels at runtime.

The H20 W8A16 comparison (~142 vs ~90 TFLOPS vs Marlin) is on a GPU that many inference operators actually run at scale (H20 is widely deployed in the Chinese market due to export restrictions). This is a meaningful throughput improvement for that hardware segment.

**Limitation:** Blackwell (SM100) support is partial; Humming is not yet the preferred backend on B200. On H100/H200/H20/A100/A800 it may be competitive or superior to Marlin for W8A16.

### 3. NIXL EP Correctness Fix Matters for FP4 MoE Deployments

PR #40412 is a silent correctness bug: NIXL EP configurations using FP4 quantization or shared experts were not being recognized as using batched-expert format. In the FP4 path, this would cause the wrong backend to be selected, potentially degrading throughput or producing incorrect numerics. The fix is prerequisite for NIXL EP + FP4 MoE to function correctly on Blackwell — which is likely an important production use case.
