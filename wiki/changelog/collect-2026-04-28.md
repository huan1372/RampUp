## Key Findings

### 1. Eagle Prefill Metadata Triple-Rebuild Eliminated (PR #40410)

The most significant performance finding today. During Eagle speculative decoding in MRV2, attention metadata was reconstructed **three times** per speculative step — once for the target model, once for Eagle prefill, once for draft decode. PR #40410 eliminates the Eagle prefill rebuild by introducing a `CapturedAttentionState` named tuple that bundles attention metadata + slot mappings for sharing.

The ~5–10% end-to-end latency improvement compounds with the v0.20.0 CUDA graph Eagle prefill capture (which reduced kernel launch overhead): that PR captured the graph; this PR removes redundant work within each captured execution. Together they represent a full optimization pass of the Eagle prefill codepath in MRV2.

**Implication:** Every deployment using Eagle or P-EAGLE speculative decoding on MRV2 (the default since v0.19.0) should see this improvement without any configuration change.

### 2. Drafter Backend Decoupling — Unlocks MLA+GQA and DFlash Drafter Configurations (PR #39930)

The forced drafter=target backend coupling was a hard blocker for configurations mixing different attention architectures. The most impactful use case: MLA-based drafters (e.g., DeepSeek-family draft models) with GQA-based targets, or vice versa. This is increasingly relevant as DeepSeek V4's popularity grows and users want to experiment with cross-architecture speculative decoding.

The key design decision — **drafter does not inherit target's backend even if target is explicitly set** — reflects the reality that incompatibilities are the common case, not the exception. This is the right default for a system with a growing matrix of attention architectures.

**Connection to DeepSeek V4 ecosystem:** DeepSeek V4 uses MLA+CSA+HCA — a hybrid attention stack that no single backend handles uniformly. PR #39930 may be a prerequisite for proper speculative decoding on V4 with cross-architecture drafters.

### 3. StepPool Double-Append: Correctness Bug for Embedding Chunked Prefill (PR #41049)

A classic off-by-one conditional scope bug. The symptom (pooling output misalignment) would manifest as silent incorrect results for embedding models processing long sequences, not as a crash — making it hard to detect in practice without explicit output shape validation.

**Scope clarification:** This bug only affects embedding/pooling models (e.g., retrieval encoders, classification heads using `step_tag_id`) using chunked prefill on long sequences. Standard text generation inference is unaffected.

### 4. TRTLLM MoE Routing Expansion (PR #39141)

Adds SigmoidRenorm and MiniMax2 routing to the TRT-LLM backend, aligned with FlashInfer v0.6.8. The MiniMax2 addition is notable as it extends TRT-LLM-accelerated serving to MiniMax-series models, which have been gaining traction. Reclassifying Custom/Simulated as internal vLLM-specific routing prevents accidental use of debug routing modes in production TRT-LLM deployments.

This is the first routing method expansion in the TRTLLM MoE backend captured in this KB.
