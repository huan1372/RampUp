## PR #40410 — [Model Runner V2] Skip Attention Metadata Rebuild Before Draft Prefill

**Merged:** April 27, 2026  
**Author:** WoosukKwon  
**Scope:** MRV2 CUDA graph pipeline, Eagle speculative decoding

### Problem

During Eagle speculative decoding in MRV2, attention metadata was being rebuilt redundantly: the pipeline rebuilt it for the target model, then again for the Eagle speculator's prefill phase, and potentially again for the draft decode. This triple-rebuild pattern imposed unnecessary GPU memory operations and kernel invocations on every speculative decode step.

### Change

PR #40410 eliminates the redundant rebuild by passing the target model's pre-built attention metadata to the Eagle speculator's prefill phase directly.

**New component:** `CapturedAttentionState` named tuple (defined in `cudagraph_utils.py`) bundles attention metadata + slot mappings into a single reusable object.

**Modified managers:**
- `PrefillEagleCudaGraphManager` — now accepts a pre-constructed `CapturedAttentionState` from the target model run; does NOT rebuild
- `DecodeEagleCudaGraphManager` — continues to build its own attention metadata (decode has different requirements)

**New workflow:** target model captures once → shares `CapturedAttentionState` with the Eagle speculator for prefill → only draft decode rebuilds independently.

### Performance

Approximately 5–10% improvement in end-to-end speculative decode latency across tested models, from eliminating GPU memory operations and kernel invocations for attention metadata construction on every speculative step.

## PR #39930 — [Attention][Spec Decode] Allow Independent Drafter Attention Backend Selection

**Merged:** April 28, 2026  
**Approvers:** fynnsu, LucasWilkinson  
**Scope:** speculative decoding configuration, attention backend selection

### Problem

The drafter model's attention backend was forced to match the target model's backend. This caused failures when draft and target had incompatible requirements:
- MLA drafter with GQA target (different head dimensions / attention structure)
- DFlash / non-causal attention needed by drafter but not supported by the target-selected backend
- Concrete failure: `ValueError: Selected backend AttentionBackendEnum.TRITON_ATTN is not valid for this configuration. Reason: ['non-causal attention not supported']`

### Change

Introduces `--speculative-config.attention_backend` as a new configuration option.

**Key design decision:** Unlike `--moe-backend`, the drafter does **NOT** inherit the target's backend when unspecified, because incompatibilities are common. Both backends auto-select independently.

**Behavior matrix:**
| target backend | drafter backend config | result |
|---|---|---|
| (auto) | (not set) | both auto-select independently |
| explicit | (not set) | target uses explicit; drafter auto-selects |
| (auto) | explicit | drafter uses explicit; target auto-selects |
| explicit | explicit | each uses its own |

### Validation

Tested on A100. Resolves the non-causal attention error for DFlash backends. Enables mixing MLA-based drafters with GQA targets.

## PR #41049 — [Core] Fix Redundant None Append in StepPool.forward for Chunked Prefill

**Merged:** April 28, 2026  
**Author:** Anthony Su  
**Approver:** noooop  
**Scope:** chunked prefill, embedding models, StepPool

### Problem

In `StepPool.forward`, a `pooled_data.append(data)` call was placed outside the conditional block checking `step_tag_id`, causing None to be appended **twice** to the output list during chunked prefill processing. For embedding models processing long sequences across multiple forward passes, this misaligned the pooling output with the batch requests.

### Change

Moves the `pooled_data.append(data)` call inside the `if step_tag_id` conditional block, ensuring None is appended only once per unfinished chunk.

### Impact

Fixes embedding model correctness for long-sequence chunked prefill (4K+ tokens). Verified on TPU with long-sequence prefill workload. No performance change — correctness fix only.

## PR #39141 — [Perf] Update TRTLLM Supported MoE Routing Methods

**Merged:** April 27, 2026  
**Merger:** robertgshaw2-redhat  
**Scope:** TensorRT-LLM backend, MoE routing, FlashInfer alignment

### Change

Updates TensorRT-LLM's supported Mixture-of-Experts routing methods to align with FlashInfer v0.6.8.

**New routing methods added:**
- `SigmoidRenorm` — sigmoid-based routing with renormalization
- `MiniMax2` — routing method used by MiniMax-series models

**Routing method reclassification:**
- `Custom` and `Simulated` routing types are now classified as internal vLLM-specific; blocked from use in external backends (TRT-LLM, FlashInfer)

**FP32 router logits:** Removes the requirement for FP32 router logits in DeepSeek MoE implementations when using the TRT-LLM backend.

### Performance

Benchmarking on MiniMax model: TRT-LLM routing implementation improves performance consistently across input/output sequence length configurations. Specific throughput numbers: "performance gains across the board" (no exact numbers in PR description).

Also covers both FP8 and NVFP4 TRTLLM MoE kernels for the new routing methods.
