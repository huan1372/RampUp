## PR #40413 — Optimize Batch Invariant with Fused RMS Norm

**Merged:** April 21, 2026
**Author:** yewentao256

**Change:** Removes a redundant conditional check inside `fused_add_rms_norm`. The PR description states: "fused_add_rms_norm is already batch invariant, we don't need to call the triton kernel [redundantly]." The underlying custom CUDA op guarantees batch invariance; the conditional was dead code that added overhead on every token.

**Performance:**
- Tested on `RedHatAI/Meta-Llama-3.1-8B-Instruct-FP8`
- Average latency: 0.524 s (main) → 0.513 s (optimized)
- **2.1% E2E latency improvement**

**Scope:** Affects all models using `fused_add_rms_norm`, which covers the majority of modern LLMs (LLaMA, Mistral, Qwen, Gemma families). No correctness risk — additional determinism tests added.

## PR #38284 — CUDAGraph Memory Profiling Enabled by Default

**Merged:** April 21, 2026
**Author:** merged by MatthewBonanni
**Originally implemented:** PR #30515

**Change:** Promotes CUDAGraph memory profiling from an opt-in environment variable to the default serving configuration. Also increases the default `--gpu-memory-utilization` from 0.9 to **0.92** to accommodate the profiling feature's overhead.

**Rationale:** Without memory profiling, complex models with high GPU memory utilization — specifically DeepSeek-R1 at DP=8 EP configurations — trigger OOM during startup. Memory profiling provides better memory estimation during CUDA graph capture, preventing these OOM errors.

**User-visible impact:**
- Default `--gpu-memory-utilization` is now **0.92** (was 0.9)
- Startup is more reliable for large MoE models in distributed configs
- Users who disable the feature via env var are warned of potential OOM risk in new log messages
- CI test suite updated with new default; 146 checks passed

## PR #38877 — MLA + Group FP8 Fusion

**Merged:** April 22, 2026
**Author:** carlyou
**Reviewer:** ProExpertProg

**Change:** Implements fusion of per-group dynamic FP8 quantization into Multi-head Latent Attention (MLA) attention kernels. Adds a new `MLAAttnFp8GroupQuantPattern` for pattern matching and updates the MLAAttention layer's forward implementation to execute quantization kernels in-kernel. Completes "phase 1 (group fp8) of issue #35792."

Also fixes NVFP4 model detection: corrects pattern matching for `DeepSeek-R1-0528-NVFP4-v2` models so that NVFP4-quantized DeepSeek variants are correctly identified and routed.

**Affected models:** DeepSeek V3 and other architectures using MLA attention with per-group FP8 quantization.

**Performance (B200x4 GPUs):**
- Output token throughput: 5,345 → 5,367 tokens/second (+0.4%)
- GSM8K accuracy: ~94.8% (maintained with fusion enabled)
- Gains are incremental — the optimization eliminates quantization bottlenecks without introducing overhead

**Background:** MLA (Multi-head Latent Attention) is DeepSeek's attention architecture that compresses KV heads through a low-rank projection, dramatically reducing KV cache size. Group FP8 applies FP8 quantization per group of heads rather than per tensor, providing more precision for MLA's per-group scale factors. Fusing these operations reduces memory bandwidth pressure.

Development span: April 3–22, 2026.

## PR #38453 — KV Offload + HMA: Multi-group Worker Transfer

**Merged:** April 22, 2026
**Reviewer:** NickLucche

**Change:** Extends the CPU-GPU KV cache offloading handler to support transfers with multiple KV cache groups. Refactors the `transfer_async` method in the CPU-GPU worker module to handle group-specific sizes and block indices. Adds support for unaligned CPU→GPU transfers across multiple groups and enhanced `expand_block_ids` with bounds checking. New test cases cover multi-group scenarios.

**Context:** vLLM's KV cache offloading system (introduced in v0.18.0/v0.19.0) stores KV blocks in CPU memory when GPU memory is exhausted, fetching them back when needed. Previously, the transfer path only handled single-group KV caches. This PR enables the offloading handler to work correctly with architectures that use multiple KV cache groups (e.g., models using MLA or multiple attention types in the same network).

**Performance:** No benchmark numbers provided in the PR. Correctness improvement only.

Development span: required multiple rebases (April 3, April 14) due to merge conflicts before landing.
