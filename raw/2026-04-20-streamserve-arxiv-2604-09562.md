# StreamServe: Adaptive Speculative Flows for Low-Latency Disaggregated LLM Serving

**arXiv**: 2604.09562  
**Submitted**: April 2026  

## Overview

StreamServe is a serving architecture that **combines disaggregated prefill-decode with runtime-adaptive speculative decoding depth**. The two key insights:

1. Disaggregation reduces head-of-line blocking but doesn't address decode latency from autoregressive generation
2. Speculative decoding can be dynamically tuned (depth K) based on live acceptance rate signals rather than a static K

## Four Components

**StreamScheduler** — orchestrates request routing and decides which decode stream (lane) each request flows through based on predicted sequence length, memory headroom, and current acceptance rates.

**FlowGuard** — multi-signal routing that monitors per-stream metrics (queue depth, KV cache pressure, TTFT SLO violation rate) to direct requests to optimal prefill-decode pairs. Acts as a dynamic load balancer across stream lanes.

**PipeServe-Engine** — the disaggregated prefill-decode execution backend. Runs prefill and decode on separate GPU pairs (each "stream pair" consists of one prefill GPU and one decode GPU). Transfers KV cache asynchronously between phases.

**SpecuStream** — the adaptive speculation component. Tunes speculation depth K online using recent acceptance rate history. High acceptance → increase K. Low acceptance → decrease K. This avoids the "wasted speculation" problem when acceptance rates fluctuate (e.g., mathematical reasoning mid-problem where token entropy spikes).

## Evaluation

Setup: 4× A800 40GB GPUs configured as 2 stream pairs (2 prefill + 2 decode). Benchmarks: ALPACA, GSM8K, HUMANEVAL, SUM — 80 queries each (320 total).

| Benchmark | Metric | TP-vLLM baseline | StreamServe | Delta |
|-----------|--------|-----------------|-------------|-------|
| GSM8K | Throughput | 241 tok/s | 264 tok/s | +9.5% |
| GSM8K | Latency | 3.50 s | 0.30 s | −91% (−3.2 s) |
| SUM | Peak throughput | — | 2235 tok/s | — |
| Overall | Latency reduction | baseline | 11–18× lower | — |

The latency improvement (11-18×) is driven primarily by disaggregation eliminating prefill-decode interference; the throughput improvement (9.5% on GSM8K) is driven by SpecuStream's adaptive speculation avoiding stale speculation budgets.

## Key Claims

- Simultaneous latency and throughput improvement is possible via architecture (not just algorithmic) optimization
- Static speculation depth K is suboptimal for reasoning tasks where acceptance rates fluctuate mid-sequence
- Disaggregation + adaptive speculation are complementary: disaggregation removes prefill-decode coupling; adaptive speculation reduces per-decode overhead

## Limitations

- Evaluated on only 320 total queries — small-scale experiment; production-scale validation not included
- Comparison baseline is tensor-parallel vLLM (no pipeline parallel, no spec decode); not compared against vLLM's native disaggregated serving with fixed-K spec decode
- A800 GPUs only; no H100/B200 results

## Relation to vLLM's Disaggregated Serving

vLLM's disaggregated serving (KV Connector API + Dynamo/llm-d routing) is a production-grade implementation of the same prefill-decode separation concept. StreamServe's SpecuStream adaptive K mechanism is not yet in vLLM — vLLM uses a fixed global K parameter per serving job. The FlowGuard routing logic overlaps conceptually with vLLM-router but adds per-stream metric feedback.
