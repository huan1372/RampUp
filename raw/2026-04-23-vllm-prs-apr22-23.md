## PR #40092 — [TurboQuant] Enable FA3/FA4 for Prefill Paths

**Merged:** April 23, 2026
**Author:** huangzhilin-hzl
**Merged by:** mgoin

**Change:** Extends TurboQuant's prefill paths to correctly select FlashAttention 3 (FA3) on Hopper (SM90) hardware and FlashAttention 4 (FA4) on Blackwell (SM100), rather than defaulting to FA2 for all architectures.

**Technical details:**
1. **FA version detection:** Added logic to detect and pass the appropriate FA version (2, 3, or 4) to `flash_attn_varlen_func` calls. Previously, TurboQuant's prefill path did not specify the FA version, causing SM90 (Hopper) to fall through to FA2 and missing FA4 entirely on SM100.
2. **Mixed-backend assert fix:** Modified `_get_sliding_window_configs()` to skip non-FlashAttention layers instead of asserting all layers are `FlashAttentionImpl`. This prevents failures when a model routes some layers to TurboQuant and others to standard FlashAttention (mixed-backend scenario).
3. Removed `--enforce-eager` flags from TurboQuant evaluation configs, as FA3 now operates reliably with CUDAGraph on TurboQuant.

**Performance (NVIDIA H20, SM90 / Hopper, using FA3):**

| Workload | Throughput improvement | TTFT reduction |
|----------|------------------------|----------------|
| prefill_heavy | +71–89% | −43–54% |
| long_balanced | +46–58% | −62–63% |

**Significance:** TurboQuant was previously leaving Hopper throughput on the table by running FA2 instead of FA3 for all prefill paths. This fix unlocks the full FA3 kernel performance for TurboQuant users on H100/H200/H20 (Hopper) and FA4 on Blackwell.

## PR #35737 — [NVFP4] NVFP4 MoE Emulation Fallback for H100/MI300/MI350

**Merged:** April 22, 2026
**Author:** fxmarty-amd (Felix Marty)
**Co-author:** kylesayrs

**Change:** Adds a software emulation fallback that allows NVFP4-quantized MoE models (e.g., Qwen3-30B-A3B-NVFP4) to run on non-Blackwell hardware: NVIDIA H100 (SM90/Hopper) and AMD MI300/MI350 series.

**Technical details:**
- Adds an emulation backend that performs FP4 quantization and dequantization in software (Triton) when hardware-native FP4 TensorCores are unavailable
- Refactors `quark_moe.py` to standardize on `TritonExperts` for OCP MX quantization emulation, removing the functional `fused_experts` dependency
- Hardware detection: automatically selects the emulation path when native NVFP4 kernels are unavailable; no user-facing flag required

**Accuracy benchmark (WikiText, Qwen3-30B-A3B-NVFP4):**

| Variant | Word perplexity | Byte-level perplexity |
|---------|-----------------|-----------------------|
| BF16 baseline | 10.90 | — |
| NVFP4 emulation | 11.26 | ~stable |

Perplexity degradation: ~3.3% relative. PIQA accuracy maintained at 0.79–0.80 across variants.

**Significance:** Previously, NVFP4-format checkpoints were hard-restricted to Blackwell GPUs. This fallback allows H100 clusters (the dominant cloud hardware as of 2026) and AMD users to load and run NVFP4 checkpoints, at the cost of running without hardware-accelerated FP4 kernels (compute-bound; slower than Blackwell native, faster than loading BF16 due to memory reduction).

## PR #40151 — [Compile] Skip FX Graph Deserialization on Loading

**Merged:** April 23, 2026
**Author:** zhxchen17
**Context:** Builds on prior work in PR #38657

**Change:** Eliminates FX graph deserialization overhead during model warm compilation by using "recently-added Python execution code as the source of truth to run compiled function moving forward" rather than deserializing graph structures at runtime. Attention submodules are inlined as Python functions rather than maintained as serialized graph representations.

**Performance (warm compile time, seconds):**

| Model | Before (s) | After (s) | Reduction |
|-------|-----------|----------|-----------|
| DeepSeek-V3.2 | 6.05 | 0.27 | −95.5% |
| GLM-4.7-FP8 | 7.07 | 0.46 | −93.5% |
| GPT-OSS-120B | 1.57 | 0.19 | −87.9% |
| Llama-3.3-70B | 3.95 | 0.20 | −94.9% |
| Qwen3.5-35B | 2.91 | 1.36 | −53.3% |

Result: sub-2-second warm compile times across most major models.

**Significance:** Warm compile (reusing a previously compiled model after a restart) is a critical path for production serving restarts and scaling events. A 6-second warm compile for DeepSeek-V3.2 becomes 0.27 seconds — this reduces restart latency for large models from seconds to sub-second.
