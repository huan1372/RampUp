## PR #40810 — [EPLB] Fix replica selection bias in fused_moe router

**Merged:** April 24, 2026  
**Author:** arpera

### Problem
The EPLB (Expert Parallel Load Balancing) fused MoE router had a deterministic hash-collision bug: when `top_k` (number of active experts per token) was a multiple of the replica count, all tokens for those experts mapped to the same replica. The resulting load imbalance exceeded 90% in adversarial configurations — i.e., one replica received 90%+ of all tokens while others were nearly idle.

### Fix
Replaced the existing hash function with a **Knuth multiplicative hash** for replica selection. Knuth's multiplicative hashing distributes keys more uniformly across hash buckets regardless of the algebraic relationship between key values and bucket count, breaking the periodicity that caused collapse.

### Results
Tested on **Qwen3.5-A17B** on **8× B200** hardware:
- **Before:** Max/mean workload ratio = **1.2** (with severe per-replica imbalances >90% in some runs)
- **After:** Max/mean workload ratio = **1.07**

New regression test: `test_eplb_map_hot_expert_replica_balance` (all 496 tests passing).

### Impact
This is a **correctness + performance fix** for any deployment using EPLB with EP ≥ 2 and `top_k` values that are multiples of the replica count. Common MoE configurations (top_k=2 with 2 replicas, top_k=8 with 4 replicas) are affected. Without the fix, EPLB degrades to near-worst-case load distribution, negating its load-balancing purpose.

## PR #34556 — [Quantization] Add Humming Quantization Kernel

**Merged:** April 24, 2026  
**Author:** jinzhen-lin (inclusionAI)

### What is Humming?
Humming is a **JIT quantization kernel library** from the inclusionAI project. It provides a flexible runtime compilation mechanism for quantization kernels, supporting a broader weight-activation format matrix than existing vLLM kernels (Marlin, CUTLASS).

### Format Support
- **Weight-activation combos:** W{1–8} A{16/8/4} — INT1 through INT8 weights, FP16/INT8/INT4 activations
- **Quantization schemes:** GPTQ, AWQ, FP8, MXFP4, NVFP4, BITNET
- **Exclusions:** GPTQ with `desc_act=True`, HQQ

### Hardware Support
- **Optimal:** Turing (SM75), Ampere (SM80), Ada Lovelace (SM89), Hopper (SM90/SM90a)
- **Partial:** Blackwell datacenter (SM100) and consumer GPUs — functional but not performance-optimized

### Online Quantization
Humming supports **online (uncalibrated) quantization** for INT1–INT8 and FP3–FP8 at serving time. Bit depths ≥ 6 recommended for uncalibrated use. Configured via environment variables; no checkpoint pre-quantization required for supported formats.

### MoE and CUDA Graph Support
- Block-wise MoE operations supported
- CUDA graph compatible

### Benchmarks
Benchmarked on H20, H200, A800/A100. Representative result:

| Configuration | Humming | Marlin | Notes |
|--------------|---------|--------|-------|
| H20, W8A16, dense, float16 activations | ~142 TFLOPS | ~89–90 TFLOPS | Large matrix dims |

Approximately **1.58× throughput advantage over Marlin** on H20 for W8A16 at large batch sizes.

### Significance
Humming is a **fourth quantization kernel backend** in vLLM, alongside Marlin (W4/W8), CUTLASS (FP8/FP4/MoE), and FlashInfer. Unlike the others, Humming is JIT-compiled, making it adaptable to new formats without kernel recompilation. The W1–W8 coverage is the broadest weight-bit range in any single kernel library currently integrated into vLLM.

## PR #40412 — fused_moe: Treat NIXL EP as Batched Experts

**Merged:** April 24, 2026  
**Author:** itayalroy

### Problem
NIXL EP (NIXL-based Expert Parallelism, the high-performance cross-node MoE routing backend using the Nixl KV transfer library) follows the **batched-expert activation path** in the fused MoE kernel, but portions of the fused MoE configuration and FP4 oracle selection only checked for `DeepEP LL` kernels when deciding whether to use batched-format handling. NIXL EP was silently excluded from:
- Activation format selection (batched vs. non-batched)
- Shared-expert handling
- FP4 backend selection

### Fix
- NIXL EP incorporated into all three batched-format check sites
- New property `needs_round_robin_routing_tables` introduced to semantically separate routing-table requirements from output-format concerns — enables finer-grained per-backend configuration

### Impact
Correctness fix rather than a new optimization. NIXL EP configurations using FP4 or shared experts were falling back to non-batched paths silently, potentially degrading throughput or producing incorrect outputs for FP4 MoE models using NIXL. After this fix, NIXL EP and DeepEP LL are treated equivalently in the configuration logic.

## PR #38503 — [ROCm][Engine] Fix GPU Memory Leaks in Engine Shutdown

**Merged:** April 25, 2026  
**Author:** AndreasKaratzas

### Problem
On **ROCm (AMD GPU)** platforms, GPU memory was not fully released on engine shutdown, causing memory leaks visible after repeated engine restarts within the same process. This affected deployments that dynamically spin up/tear down vLLM engines.

### Fix
Memory release ordering corrected at ROCm engine shutdown; a test workaround added for CI isolation.

### Impact
Relevant for AMD MI300X/MI350X deployments. No impact on NVIDIA (CUDA) paths. Low inference-optimization significance, but prevents resource exhaustion in multi-tenant or restart-heavy deployments.

## PR #40844 — [Bugfix] Add seq_lens_cpu_upper_bound to CommonAttentionMetadata in mla_runner.py

**Merged:** April 24, 2026  
**Author:** ignaciosica

### Problem
The MLA runner's `CommonAttentionMetadata` was missing `seq_lens_cpu_upper_bound`, a field needed for correct upper-bound computation of CPU-side sequence lengths during MLA attention. This could cause incorrect memory allocation in edge cases (sequences near the upper length boundary).

### Fix
Added `seq_lens_cpu_upper_bound` to `CommonAttentionMetadata` in `mla_runner.py`.

### Impact
Correctness fix for MLA (Multi-head Latent Attention) models (DeepSeek V3, V4, etc.) in edge-case sequence length scenarios. Complements PR #40654 (seq_lens_cpu sync removal) from v0.20.0.

## PR #40715 — [BE][Bugfix] Respect TORCH_COMPILE_DISABLE Env Var at vLLM Config Level

**Merged:** April 24, 2026  
**Author:** Lucaskabela

### Summary
When the `TORCH_COMPILE_DISABLE=1` environment variable was set, vLLM's backend configuration was not honouring it — compilation would still proceed for certain code paths. This PR propagates the flag correctly through the vLLM config layer.

### Impact
Useful for debugging and performance profiling: `TORCH_COMPILE_DISABLE` lets operators isolate compilation overhead from runtime overhead. Now reliable across all vLLM config paths.

## PR #40797 — [Feature] Warm Up Readonly Multimodal Processor During Renderer Startup

**Merged:** April 25, 2026  
**Author:** fake0fan & codex

### Summary
The readonly multimodal processor (used for image/video tokenization in multimodal models) is now warmed up during renderer startup rather than on the first request. This eliminates a latency spike (JIT compilation + model load) on the first multimodal request to a cold server.

### Impact
Reduces TTFT for the **first** multimodal request. No throughput change for subsequent requests. Relevant for deployments serving multimodal models (Qwen2-VL, LLaVA, etc.) where cold-start latency matters.
