## PR #40941 — [Attention][TurboQuant] Share dequant buffers, eliminate float16_copy

**Merged:** April 27, 2026
**Author:** bhoomit

### Problem

TurboQuant's KV cache attention backend allocated per-layer dequantization buffers that grew with context length. At 1M tokens (32B model, TP=4), these buffers consumed **58.6 GB** of GPU memory — rendering 1M-context TurboQuant inference impractical. Secondary issues: redundant `.zero_()` calls, an unnecessary `float16_copy` kernel in decode, and `.contiguous()` copies during continuation prefill.

### Technical Approach

Seven optimizations applied:

1. **Shared dequant buffers via WorkspaceManager** — shifts buffer allocation from per-layer to a global workspace pool; enables reuse across transformer layers, capping peak memory to the size needed for one layer
2. **Eliminated redundant `.zero_()` calls** — buffers are written before read in all paths; zeroing is unnecessary overhead
3. **FP16 Hadamard rotation** — matrix multiplication performed in float16 instead of casting to float32; reduces memory bandwidth
4. **Pre-sized continuation buffers** — K/V tensors pre-allocated to prevent allocation storms during continuation prefill
5. **Cached utility tensors** — `arange` and sequence-length tensors pre-built for speculative decoding paths
6. **In-kernel dtype casting via `OUTPUT_FP16` constexpr** — stage2 kernel modified to cast output dtype inline, eliminating a separate copy kernel
7. **dtype mismatch fix** — corrects shared-buffer decode path to use query dtype instead of a fixed type

### Performance Results

**Memory savings (32B model, TP=4):**
| Context | Memory saved |
|---------|-------------|
| 8K | 472 MB |
| 128K | 7.4 GB |
| 1M | **57.6 GB** |

**Throughput:** Within noise margins in both eager and CUDA Graph modes; decode throughput remained stable at ~44 tok/s. 117 existing unit tests pass.

### Impact

- Makes long-context TurboQuant inference (128K–1M tokens) practically feasible — previously would OOM
- Enables CUDA Graph capture compatibility for TurboQuant (CUDA graphs require pre-allocated fixed-size buffers; per-layer variable-size buffers blocked this)
- No quality impact: 117 unit tests pass

## PR #40950 — [DSV4] Add silu clamp limit to shared expert

**Merged:** April 27, 2026
**Author:** zyongye

### Problem

DeepSeek V4's shared expert MLP uses a SiLU (`silu_and_mul`) activation function. Without clamping, extreme activation values caused numerical instability during inference — a correctness issue manifesting as incorrect outputs in edge cases with high-magnitude hidden states.

### Technical Approach

- Introduces a new `DeepseekV4MLP` class for the shared expert with clamping parameters
- Clamp limits are **baked into the activation kernel** (`silu_and_mul` CUDA kernel modified with a `clamp_limit` parameter) rather than applied as a separate post-processing tensor op
- The kernel-level approach eliminates an extra memory round-trip vs a separate clamp pass

### Impact

- **Correctness fix**: prevents numerical overflow/underflow in DeepSeek V4's shared expert at extreme hidden-state magnitudes
- Minor throughput improvement from eliminating a separate clamp tensor operation
- Affects DeepSeek-V4, DeepSeek-V4-Flash, and any future models using `DeepseekV4MLP`

## PR #38065 — [Perf] FP8 FlashInfer Attn for ViT

**Merged:** April 27, 2026
**Authors:** zhandaz, ybgao-nvidia

### Problem

In multimodal workloads (image + text inference), the Vision Transformer (ViT) encoder attention becomes a significant bottleneck when the text model is already quantized. ViT attention was running in BF16/FP16 while the language model ran in FP8 — the ViT was relatively more expensive, especially at high image resolutions.

### Technical Approach

Adds optional FP8 quantization for ViT encoder attention using the FlashInfer cuDNN attention backend:
- **Backend:** FlashInfer cuDNN (not standard FlashAttention)
- **Scaling:** Dynamic scaling by default; optional static calibration for production
- **Hardware gate:** Requires SM90+ (Hopper or Blackwell); no-op fallback on older architectures
- **cuDNN minimum version:** 9.17.1

### Supported Models

Qwen3 VL family only (as of merge date):
- `qwen3_vl`, `qwen3_vl_moe`, `qwen3_5`, `qwen3_5_moe`

### Performance Results

**E2E Encoder Forward Time (Qwen3-VL-30B, GB200, 3 images/request):**
| Image resolution | Speedup |
|-----------------|---------|
| HD (720×1280) | 0.87× (regresses) |
| FullHD (1080×1920) | 0.99× (neutral) |
| QHD (1440×2560) | 1.08× |
| 4K (2160×3840) | **1.18×** |

**Core cuDNN kernel speedup (head_dim=128, seq_len=8192):**
| Hardware | Speedup |
|----------|---------|
| GB200 | 1.12× |
| GB300 | **1.42×** |

**Key pattern:** FP8 ViT attention is only beneficial at high resolutions (high sequence lengths in the vision encoder). At HD resolution, the overhead of FP8 conversion exceeds the kernel speedup.

### Impact

- Meaningful throughput gain for 4K/QHD multimodal workloads on Blackwell/Hopper hardware
- No benefit (slight regression) for low-resolution images — configurable, not forced
- GB300 sees larger gains, suggesting Blackwell B200's ViT attention kernel is less bandwidth-bound

## PR #40946 — [Bugfix] Cap SWA/chunked-local runtime admission to startup pool-sizing bound

**Merged:** April 27, 2026
**Authors:** Dao007forever, njhill

### Problem

On models with **hybrid full + sliding-window attention (SWA)** or **chunked local attention**, the scheduler had a formula mismatch between two places that estimate KV block requirements:

1. **Startup pool sizer** — accounts for block recycling via `SlidingWindowManager.remove_skipped_blocks` (out-of-window blocks are freed between chunks during chunked prefill)
2. **Runtime admission gate** — did not account for this recycling; used a simpler formula that overestimated required blocks

When the KV pool was sized at the startup minimum (pool exactly large enough for recycling-aware worst case), the runtime gate's overestimate caused it to reject long prompts that would actually fit, creating a **scheduler deadlock** — requests were perpetually rejected even when sufficient memory existed.

### Fix

Introduces a unified `max_admission_blocks_per_request` method on both `SlidingWindowSpec` and `ChunkedLocalAttentionSpec`. Both the startup pool sizer and the runtime admission gate now call this single method — eliminating formula drift.

Additional changes:
- Threads `max_num_batched_tokens` through KV cache coordination layers so the admission gate can use the same recycling-aware calculation
- Caps demand in `get_num_blocks_to_allocate` using the recycling-aware bound
- Tests confirm: long prompts are admitted on capped pools; genuinely oversized requests are still correctly rejected

### Impact

- **Correctness fix**: eliminates scheduler deadlock on SWA/chunked-local models when pool is sized at minimum
- Affects models using sliding window attention (e.g., Mistral, Gemma variants with SWA) or chunked local attention with long prompts
- No performance change — correctness only

## PR #40346 — [KV Offload] Offload all KV blocks when doing prefill in P/D

**Merged:** April 26, 2026
**Authors:** omerpaz95, orozery

### Problem

In disaggregated prefill/decode (P/D) serving where the prefill node sends the full KV cache to a remote decode node (`do_remote_decode=True`), the KV cache OffloadingConnector only offloaded **new blocks** since the last store (tracked by `next_stored_block_idx`). For the remote decode use case, the decode node needs the **complete** KV cache, not just the incremental delta.

### Fix

Resets `start_block_idx` to 0 when `kv_transfer_params["do_remote_decode"]` is enabled. This causes the full KV cache to be exported to CPU memory for the remote decode node to consume.

**No-op with local backends:** When combined with the MultiConnector and local CPU backends, backends automatically detect already-cached blocks and avoid re-copying — no performance penalty on the local path.

### Impact

- Enables end-to-end P/D disaggregated serving where the CPU offload layer is the transport medium for cross-node KV transfer
- Prerequisite for deployments using CPU-backed KV transfer (vs direct GPU-to-GPU NVLink/RDMA paths)
- Minimal scope: surgical one-field change, only active when `do_remote_decode=True`
