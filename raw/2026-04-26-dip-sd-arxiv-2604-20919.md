# DiP-SD: Distributed Pipelined Speculative Decoding for Efficient LLM Inference at the Edge

**arXiv ID:** 2604.20919
**Authors:** Yaodan Xu, Sheng Zhou, Zhisheng Niu
**Submitted:** April 22, 2026 (appeared on arXiv April 24–25, 2026)
**Venue:** Not yet peer-reviewed

---

## Problem

Standard speculative decoding (SD) assumes a co-located draft model and target model on the same GPU server. In **edge inference** scenarios, this assumption breaks:

- Multiple users have small LLMs on their **end devices** (phones, tablets) — these can serve as draft models
- A centralized **edge server** hosts the large target model for batch verification
- Drafting and verification are now **distributed across the network**, with non-trivial communication latency

The challenge: jointly optimize (a) which users to batch together, (b) how many speculative tokens K each user drafts per iteration, and (c) how to pipeline draft generation with server-side verification — all under coupled latency, memory, and communication constraints.

---

## Approach: Two Complementary Parallelism Dimensions

DiP-SD exploits two orthogonal dimensions not available in co-located SD:

### 1. Device-Level Distributed Drafting
Multiple user devices generate draft tokens in parallel. Rather than a single draft model producing K tokens sequentially, N users each draft K tokens simultaneously. From the server's perspective, batch verification amortizes the target model cost across N concurrent users.

### 2. Phase-Level Draft-Verify Pipelining
Draft generation on devices and verification on the edge server are **pipelined** — the server verifies batch i's drafts while the users generate drafts for batch i+1. This eliminates the sequential dependency between phases (draft → send → verify → return) that creates idle time in naive distributed SD.

### Joint Optimization
Key control variables are co-optimized:
- **User-to-batch assignment**: which users are batched per verification round
- **Per-user speculative length K**: how many tokens each user drafts (can vary per user based on their device capability and acceptance rate history)
- **Number of pipeline batches**: controls memory/throughput tradeoff on the server

Optimization objective: maximize **expected accepted tokens per unit time** (throughput-equivalent metric) subject to per-user latency constraints.

---

## Setting

DiP-SD targets the **multi-user edge inference** scenario:
- Draft model: on-device small LLM (deployed on phone/tablet)
- Target model: edge server (not cloud — ~10ms RTT vs ~100ms for cloud)
- Communication: device-to-server transmission of draft tokens; server-to-device return of accepted prefix
- Paper's claim: "first SD scheme to exploit batch-level pipeline parallelism between distributed drafting and centralized verification"

This is architecturally distinct from datacenter speculative decoding (P-EAGLE, CSD, SMC-SD) where all computation is on-server.

---

## Results

From the search results, the paper states "extensive numerical experiments demonstrate superior performance across all test points" compared to:
- Naive distributed SD (no pipelining)
- Greedy batching without joint K optimization

Specific throughput speedup numbers were not captured in accessible web content (arXiv HTML page returned 403). The paper optimizes for edge throughput (accepted tokens/sec/server) rather than per-user TTFT.

---

## Relation to vLLM

DiP-SD is **not integrated into vLLM** and is unlikely to be in its current form — vLLM assumes co-located draft/target models. The edge setting (device draft + server verify) requires a fundamentally different execution model.

However, the batching and pipelining insights are relevant to vLLM's existing SD infrastructure:
- vLLM's static K per job could benefit from per-request K tuning (analogous to DiP-SD's per-user K)
- The pipeline parallelism between draft and verify steps maps loosely to vLLM's async scheduling

## Open Questions

- What is the quantitative speedup (vs naive distributed SD and vs greedy batching) in the paper's experiments?
- What on-device models were used (model size, architecture)?
- Does the DiP-SD framework assume a fixed acceptance rate per user, or does it track live acceptance rates?
- How does DiP-SD handle heterogeneous user devices with different draft model sizes and generation speeds?
