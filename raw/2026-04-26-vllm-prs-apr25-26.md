## PR #40893 — [Bugfix] Size FlashInfer NVLink MNNVL workspace to EP group

**Merged:** April 26, 2026
**Author:** Dao007forever

### Problem
The FlashInfer NVLink managers (`FlashInferNVLinkOneSidedManager` and `FlashInferNVLinkTwoSidedManager`) were sizing their MNNVL (Multi-Node NVLink) workspace allocation to the **Data Parallel (DP) group size** rather than the **Expert Parallel (EP) group size**. When `dp_size != ep_size`, both managers raised assertion failures because the kernel expected workspace dimensions matching the EP group.

- **One-sided path**: kernel asserts `workspace.size(0) == moe_ep_size` — received DP-sized allocation
- **Two-sided path**: similar mismatch via `MnnvlMemory.set_comm_from_config`

These managers only exist on EP communicators (gated in `base_device_communicator.py`), so `self.cpu_group` always represents the EP group, regardless of DP/PCP/TP configuration.

### Fix
Both managers updated to use `self.cpu_group` (EP group) instead of `get_dp_group().cpu_group` (DP group) for workspace sizing.

### Results / Impact
- **Correctness fix**: assertion failure resolved for any DP+EP configuration with `dp_size != ep_size`
- **Tested on:** Kimi-K2.5-NVFP4 with TP=2, DP=4, EP configuration
- **Affected:** Any deployment using FlashInfer NVLink (Blackwell multi-node) with DP>1 and EP>1 where `dp_size != ep_size`

## PR #40865 — [Bugfix][MoE] Only unpad routed output before shared expert add or routed output transform

**Merged:** April 25, 2026
**Authors:** netanel-haber, jeejeelee

### Problem
The MoE runner was unconditionally unpadding (truncating) the routed expert output before shared-expert addition. This caused failures in MoE architectures that **lack shared experts** (or lack a routed output transform), where the premature truncation produced non-contiguous tensor issues and incorrect results.

### Fix
Added a conditional check: unpadding is only performed when `self.has_shared_expert or self.routed_output_transform is not None`.

### Impact
- **Correctness fix**: restores GPT-OSS model functionality on B200 hardware (GPQA eval was crashing)
- Preserves the behavior of prior fixes for Nemotron-Nano-v3 (TP=1, shared-expert scenario)
- **Affected models**: GPT-OSS 20B (B200), Nemotron-Nano-v3, and other latent MoE models with routed output transforms

## PR #39403 — [kv_offload+HMA][11/N]: Support store with multiple KV groups

**Merged:** April 25, 2026
**Author:** orozery

### Problem
vLLM's KV cache offloading connector assumed a **single KV cache group** when computing offloadable tokens and managing block indices in `build_connector_meta`. Architectures that use **multiple KV groups** — specifically **Hierarchical Multi-token Attention (HMA)** models — could not use the KV offloader because the single-group path did not generalize.

### Fix
Refactors the offloading scheduler's `build_connector_meta` to:
- Calculate offloadable tokens across multiple KV groups
- Manage block indices per group during distribution
- Handle block storage logic with group-specific partitioning

### Limitations
- Does not support sliding window attention or SSMs (State Space Models) — scoped to standard attention patterns
- Infrastructure PR (part 11 in a series for HMA KV offload support)
- No benchmark numbers — correctness/capability extension only

### Impact
Enables KV cache offloading for HMA-class models in vLLM. This is a prerequisite for KV offloading to work with hierarchical attention architectures like DeepSeek V4's CSA/HCA KV layout. Reviewed and merged by maintainer markmc.

## PR #40806 — [Bugfix] Fix DSML token leakage in DSV4/3.2

**Merged:** April 26, 2026
**Authors:** chaunceyjiang, sfeng33

### Problem
During **streaming tool calls** with DeepSeek V4 and V3.2 models, the DSML (DeepSeek Markup Language) sentinel token `｜DSML｜` was leaking into streamed content when the marker spanned response chunk boundaries. The output parser was emitting the partial marker before confirming it was a complete sentinel — causing garbled output in tool-calling scenarios.

### Fix
- New `_extract_content()` method using partial tag overlap detection
- Tracks emission position via a `_sent_content_idx` cursor per request (pattern from KimiK2 and Glm4Moe parsers)
- Buffers any text suffix that could form a prefix of `<｜DSML｜tool_calls>`

### Impact
- **Correctness fix** for streaming tool-call reliability on DeepSeek V4, V4-Flash, V3.2 models
- No throughput or latency impact (pure buffer boundary logic)
- Comprehensive test coverage added for character-by-character streaming and all interior split points
