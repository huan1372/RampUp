### PR #37463 — [Kernel] Add MXFP4 W4A4 CUTLASS MoE kernel for SM100

**Topic**: New CUTLASS kernel for Mixture-of-Experts inference using MXFP4 format with W4A4 (4-bit weights, 4-bit activations) on NVIDIA Blackwell B200 (SM100 architecture).

**Technical details**:
- Introduces a CUTLASS-based MoE expert computation kernel targeting SM100 (Blackwell B200)
- Uses MXFP4 (microscaling FP4) format for both weights and activations — the most aggressive quantization now in vLLM
- W4A4 means weights stored at 4-bit and activations quantized to 4-bit during the forward pass
- CUTLASS (CUDA Templates for Linear Algebra) backend is an alternative to the existing FlashInfer MXFP4 MoE kernel
- SM100 introduces hardware-native FP4 TensorCores; this kernel is designed to exploit them directly via CUTLASS
- Uses ThreadBlockShape = Shape<_128, _128, _128> and TMA (Tensor Memory Accelerator) integration
- Specifically targets MoE (Mixture of Experts) expert gating computation — the batched matrix multiply across experts
- Complements existing NVFP4 non-gated CUTLASS MoE support (PR #37320)

**Context**:
- vLLM's Blackwell SM100 MoE support previously relied on FlashInfer backends with MXFP4/MXFP8
- The Hugging Face/apsys TFLOPS analysis showed that kernel quality matters enormously for FP4 MoE on Blackwell — native FP4 TensorCores deliver far more TFLOPS than FP8 fallbacks
- MXFP4 achieves ~4× weight memory reduction vs BF16; W4A4 additionally reduces activation memory during computation
- SM100 hardware: B200 GPU, part of NVL72 rack or HGX configurations; 192 GB HBM3e per GPU

**Status**: Merged into vLLM main, targeting upcoming release (not yet in a numbered release as of April 19, 2026).

### PR #40194 — [Attention] TurboQuant: remove redundant random signs, add prior art attribution

**Topic**: Refinement to the TurboQuant KV cache compression implementation (originally merged PR #38479, April 15, 2026).

**Technical details**:
- Removes redundant random sign flip step from the WHT rotation pipeline
  - WHT (Walsh-Hadamard Transform) theoretically does not require random sign flipping when applied as a channel decorrelation transform; the original implementation included it unnecessarily
  - Removing it reduces per-token KV quantization overhead without changing compression quality
- Adds attribution to prior academic work on WHT-based quantization (TurboQuant ICLR 2026 paper, Google Research)
- No change to the compression presets (turboquant_k8v4, turboquant_4bit_nc, turboquant_3bit_nc, tq_k4v3) or their behavior

**Significance**: Performance improvement for TurboQuant — reduces the compute overhead of the WHT step, which was one of the cited sources of short-context throughput regression. The change should modestly improve TPOT at all context lengths for TurboQuant presets.

### PR #40143 — [Core] Reduce mm scheduler, get_num_embed overhead

**Topic**: Multimodal (MM) scheduler efficiency — reduces overhead from `get_num_embed` calls in the scheduler hot path.

**Technical details**:
- `get_num_embed` computes the number of embedding tokens for multimodal inputs (images, audio) at scheduling time
- Called repeatedly in the scheduler loop to check if a request fits in the current batch
- PR reduces overhead by caching or short-circuiting redundant calls
- Directly reduces the scheduling latency for vision-language workloads (LLaVA, Gemma4 vision, InternVL, etc.)
- Complements the zero-bubble async scheduler (default in v0.19.0) — with async scheduling, scheduler time is on the critical path overlap

**Significance**: Useful for operators serving multimodal models where the scheduler overhead was measurable at high request rates.

### PR #39185 — [KV Offload] Pass request context

**Topic**: KV cache offloading correctness — propagates request context information across offloading subsystem.

**Details**: Ensures per-request metadata (e.g., lora adapter, request priority, prefix hash) is correctly passed into the KV offloading path. Without this, certain request-specific cache decisions (e.g., which prefix blocks to prioritize for GPU vs CPU) were made without full context. Correctness fix, not a throughput improvement.
